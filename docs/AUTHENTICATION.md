# Authentication — Tokens, Refresh & Revocation

How a session is established and kept secure: a short-lived access JWT, a rotating opaque refresh token, and
immediate revocation via SecurityStamp. Authorization (roles, permissions) is separate — see
[PERMISSIONS.md](PERMISSIONS.md). Endpoint listings: [ARCHITECTURE.md](ARCHITECTURE.md).

**Scope:** `Modules.Identity` (API) + `core/auth` (Portal).

## Token model

| Token | Form | Lifetime | Storage |
|---|---|---|---|
| **Access** | HS256 JWT | 15 min (config) | in-memory in the SPA (never `localStorage`) |
| **Refresh** | opaque 256-bit random (not a JWT) | 14 days (config) | httpOnly cookie; only its SHA-256 hash is persisted |

- **Access token** claims: `sub`, `domainUserId`, `is_admin`, `email`, `phone`, plus a **`stamp`** claim carrying the user's ASP.NET Identity `SecurityStamp`. Generated by `TokenService.GenerateToken`. The tenant role is **not** in the token — it is resolved per request from the DB.
- **Refresh token** carries no claims; it is meaningful only as a DB row. The raw value is returned once (as a cookie); only its hash is stored.

Tunables: `Jwt:AccessTokenMinutes` (15), `Jwt:RefreshTokenDays` (14), `Jwt:RefreshTokenCleanupIntervalHours` (6),
`Jwt:RefreshTokenRetentionDays` (1).

## Data model — `RefreshToken`

`Modules.Identity/Infrastructure/Identity/RefreshToken.cs`, table `RefreshTokens` on the **Identity** migration
chain.

| Column | Purpose |
|---|---|
| `Id` | PK |
| `UserId` | AppUser id — matches the JWT `sub` |
| `TokenHash` | SHA-256 of the raw token; **unique index**. Raw never stored |
| `FamilyId` | rotation lineage — all tokens descended from one login share it |
| `CreatedAtUtc` / `ExpiresAtUtc` / `RevokedAtUtc` | lifecycle timestamps |
| `ReplacedByTokenId` | set on rotation → marks the token "spent" (reuse signal) |
| `CreatedByIp` | audit |

## Rotation + reuse detection

`RefreshTokenService` owns the lifecycle:

1. **Login / Register** → `IssueAsync` creates a token at the head of a fresh **family**.
2. **Refresh** (`POST /api/auth/refresh`) → `ValidateAsync` then `RotateAsync`:
   - unknown / expired / revoked → reject.
   - **already-spent (replayed)** → reuse detected: revoke the **entire family** and reject. A stolen token and the victim's token can't both survive — the next use by either trips this.
   - active → mark spent, issue a successor in the same family, return a new access + refresh pair.
3. **Logout** (`POST /api/auth/logout`) → revoke the presented token's family, clear the cookie.

## Revocation — two tiers

**Tier 1 — refresh denial (always on).** Because access tokens live ~15 min, revoking refresh rows caps exposure
at the access-token lifetime. Covers ordinary logout and "log out everywhere".

**Tier 2 — SecurityStamp check (all users).** Every authenticated request re-validates the access token's `stamp`
claim in `JwtBearerEvents.OnTokenValidated` against the user's current `SecurityStamp`, cached in shared
`IDistributedCache` (Redis in Production) for 60s to avoid a per-request DB hit. Rotating the stamp invalidates
**all** live access tokens for that user. Triggers:

- `POST /api/auth/logout-all` (`RevokeAllSessionsCommand`) — revokes all refresh tokens **and** rotates the stamp.
- **Password change / reset** — Identity rotates the stamp automatically; handlers also revoke refresh tokens and evict the cached stamp.
- **Platform-admin grant/revoke** (`PromoteUserToAdminCommand`) — on an actual status change, rotates the stamp and evicts the cached one. Because `is_admin` rides in the access token, a demotion revokes the former admin's live tokens on their next request rather than waiting ≤15 min for expiry.

Worst-case revocation latency for a still-cached stamp is 60s; the operations above call
`ISecurityStampCacheService.EvictAsync`, so the next request on any instance misses cache, loads the current stamp,
and rejects revoked tokens immediately. Without Redis (Development only), eviction is local to the process.

## Endpoints

| Method | Route | Auth | Command |
|---|---|---|---|
| POST | `/api/auth/refresh` | cookie | `RefreshTokenCommand` |
| POST | `/api/auth/logout` | cookie | `LogoutCommand` |
| POST | `/api/auth/logout-all` | `[Authorize]` | `RevokeAllSessionsCommand` |

`login`/`register` also set the refresh cookie; their response body is `{ token }`.

**Cookie:** `gymbro_refresh` — `HttpOnly`, `SameSite=Strict`, `Path=/api/auth`, `Secure` when the request is HTTPS
(so http localhost works in dev). Scoped to `/api/auth` so it never rides along on ordinary API calls.

## Frontend (`GymBroPortal/src/app/core/auth`)

- **`auth.ts`** — access token in an in-memory signal. On app start the constructor runs a **silent refresh** against the cookie to restore the session. `refresh()` is **single-flight** so concurrent 401s cause one network call. Exposes `ready` (a promise) so guards can wait for the initial refresh.
- **`error-interceptor.ts`** — on a 401 from a non-auth call, silently `refresh()` then replay the request once; if refresh fails, `logout()`. Auth endpoints are exempt to avoid recursion.
- **`auth-guard.ts`** — awaits `auth.ready` before deciding, so a reload doesn't bounce a logged-in user.

## Background cleanup

`RefreshTokenCleanupService` (hosted `BackgroundService`) purges rows past expiry every
`Jwt:RefreshTokenCleanupIntervalHours`, keeping `Jwt:RefreshTokenRetentionDays` of grace. Spent/revoked-but-unexpired
rows are retained so reuse detection still works.

## Security properties

- Refresh token is 256-bit CSPRNG, stored hashed → a DB dump yields no usable tokens. httpOnly + in-memory access token keeps the long-lived credential out of XSS reach.
- Reuse detection turns token theft into a self-revoking event.
- `SameSite=Strict` + the custom `X-Tenant-Id` header give a solid CSRF posture on the refresh endpoint; add an explicit anti-CSRF token if `SameSite` is ever relaxed.
- HS256 is sufficient for the monolith (symmetric key, single verifier). The signing key is rotated via secret management, never committed; startup rejects a missing/short/placeholder `Jwt:Secret`.

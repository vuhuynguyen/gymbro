## Findings (detail)

### 1. [HIGH] `PrCount` and detail `IsPr` use different prior-best scopes
- **Category:** Conflicting business rule / read-model drift.
- **Root cause:** `CompleteSessionHandler.ComputePrCountAsync` computes the prior best from tenant-scoped
  `sessionRepository.Query()` (gym-local), while the trainee detail view (`GetMyWorkoutSessionByIdHandler`) and the
  Progress page compute it from `QueryOwnAcrossGyms` (lifetime). `BUSINESS_RULES.md` defines `PrCount` as "across all
  earlier sessions" â€” i.e. lifetime.
- **Impact:** A trainee who trains in two gyms sees a session badged "1 PR" in the list but "0 PRs" in the detail.
- **Fix:** Compute the prior best from the cross-gym own-history source so the stored read-model matches the live
  recompute and the documented semantics.
- **Why superior:** One prior-best definition for the same concept; the stored count can no longer contradict the
  detail view.

### 2. [HIGH] `UserTenantRole` missing `TenantId`-leading index
- **Category:** Index gap vs. predicate.
- **Root cause:** The only index is the unique `(UserId, TenantId)`. A B-tree can't serve a predicate filtering on the
  *trailing* column, but `GetByTenantAsync` filters `TenantId == @t` and is the backbone of the members list, coach
  roster, member-count, and admin member list.
- **Impact:** Sequential scan whose cost grows with *total* memberships across all tenants, on nearly every
  authenticated tenant-scoped request.
- **Fix:** Add a non-unique `(TenantId, UserId)` index (one AppDbContext migration). Keep the unique index.

### 3. [HIGH] Unbounded pagination on `ListSessions` + catalog search
- **Category:** Pagination / DoS.
- **Root cause:** Clamping was applied per-handler by *convention*; `ListSessionsHandler`, `SearchExercises`, and
  `SearchFoods` were missed â€” they pass raw client `pageSize` to `.Take(...)`. The exercise search additionally embeds
  the unbounded page size in the cache key (cardinality blow-up).
- **Impact:** An authenticated caller issuing `pageSize=1000000` forces a huge materialization + N+1 aggregates â€” a
  cheap memory/CPU DoS.
- **Fix:** Clamp `page`/`pageSize` in the missed handlers (`Math.Clamp(pageSize,1,100)`, `Math.Max(page,1)`),
  mirroring every other list handler.
- **Why superior:** Removes the per-handler-convention footgun for the offenders; defends a public surface.

### 4. [MEDIUM] Three divergent e1RM "PR-eligible" gates
- **Category:** Conflicting business rules.
- **Root cause:** "What counts as a credible e1RM PR" is reimplemented per call-site. Epley e1RM is computed for *any*
  positive reps, so a 20-rep set yields an inflated e1RM. The Progress page gates strictly (`Working âˆ§ e1RM âˆ§ repsâ‰¤12
  âˆ§ Strength/Bodyweight`); `PrCount` and detail `DetectPrs` do **not** apply the reps cap or tracking-type filter.
- **Impact:** A high-rep set lights up a bogus "PR" on the session list/detail but is (correctly) ignored by Progress
  â€” internally contradictory reporting.
- **Fix:** Extract one `IsPrEligible(...)` predicate (in `ExerciseTrackingRules`) and use it in all three places.
- **Why superior:** Single authoritative PR-eligibility rule; the e1RM *formula* is already single-sourced, the *gate*
  now is too.

### 5. [MEDIUM] `AdminRemoveMember` can orphan a tenant
- **Category:** Invariant-guard omission + race.
- **Root cause:** The "tenant keeps â‰¥1 Owner" invariant lives only in `LeaveTenant`/`RemoveMember`. The admin path
  removes any membership (incl. an Owner) with no last-owner check and no `pg_advisory_xact_lock`, so it isn't even
  serialized against a concurrent `LeaveTenant`.
- **Impact:** An ownerless tenant: unadministrable plans/assignments; the documented "last Owner cannot leave"
  invariant is silently defeated through the admin door.
- **Fix:** Take the membership-change advisory lock and reject removing the last Owner, mirroring `LeaveTenant`.

### 6. [MEDIUM] `CompleteSession` trusts unbounded client `CompletedAt`
- **Category:** Edge case / input validation on a state transition.
- **Root cause:** `WorkoutSession.Complete` derives `DurationSeconds` from `completedAt ?? UtcNow`, and `completedAt`
  flows straight from the client command with no validator rule.
- **Impact:** A client can post `CompletedAt` before `StartedAt` (negative duration) or far-future (inflated),
  polluting history and duration analytics.
- **Fix:** Validate `CompletedAt âˆˆ [StartedAt, now+skew]`; clamp `DurationSeconds â‰¥ 0` in the entity as
  defense-in-depth.
- **Why superior:** Duration is a server-derived fact; the "never trust client time" rule already governs day
  boundaries â€” this extends it to session duration.

### 7. [MEDIUM] `apply-latest` unhandled unique-index collision
- **Category:** TOCTOU / missing conflict handling.
- **Root cause:** Re-pointing an assignment to the latest version can collide on the `(TenantId,TraineeId,PlanId)`
  live-assignment unique index when the trainee already holds a live assignment of that template at the target
  version. Unlike create/edit/publish, `apply-latest` neither pre-checks nor catches `DbUpdateException`.
- **Impact:** 500 instead of a clean 409; the coach can't tell a duplicate is the cause. (Workout + nutrition.)
- **Fix:** Catch `DbUpdateException` â†’ `Conflict`, consistent with every sibling write that guards this index.

### 8. [MEDIUM] No account lockout on password login
- **Category:** Auth / brute-force.
- **Root cause:** `LoginHandler` uses `userManager.CheckPasswordAsync`, which never increments the failed-access
  counter or honors `LockoutEnd` (the columns exist but are unused). Defense is the per-IP rate limit only.
- **Impact:** A distributed/rotating-IP attacker can attempt passwords against a known account unthrottled per-account.
- **Fix:** Use the lockout-aware check (`AccessFailedAsync`/`IsLockedOutAsync` around the password check) + lockout
  options (5 attempts / 15-min window). Keep the IP rate limit as the outer layer.

### 9. [MEDIUM] WorkoutSession handlers repeat the load-and-authorize preamble 9Ã—
- **Category:** Duplication.
- **Root cause:** The shared-lifecycle dedup covered plan/assignment *templates*; the per-session command guard
  (load â†’ 404 if missing â†’ 403 if not owner â†’ 409 if not InProgress) was never extracted, despite the team proving the
  pattern with `TraineeEditingDisabledGuard`.
- **Impact:** A change to this security-sensitive row-level contract must be made in 9 places and risks drift.
- **Fix:** Extract a guard returning the loaded aggregate or an `Error`; handlers consume it.

### 10. [MEDIUM] Three actions bypass the centralized error mapper
- **Category:** Error-handling consistency.
- **Root cause:** `UserController.GetMyTenants`, `AuthController.ForgotPassword`, `AuthController.ResetPassword`
  hard-code `BadRequest(result.Error.Message)` instead of `result.ToFailureResult(this)`, so any non-BadRequest error
  is silently coerced to 400.
- **Fix:** Route all three through `ToFailureResult`.

### 11. [MEDIUM] `GetMyTenants` counts members in memory
- **Category:** In-memory aggregation that belongs in SQL.
- **Root cause:** Loads every membership row of every tenant the caller belongs to, then `.GroupBy().Count()`
  client-side.
- **Fix:** Project the group count in SQL (`GroupBy(...).Select(g => new { g.Key, Count = g.Count() })`).

### 12. [MEDIUM] `ListSessions` materializes tracked entities
- **Category:** Missing `AsNoTracking` on a read.
- **Fix:** Add `.AsNoTracking()` (its `/api/me` twin already does).

### 13. [MEDIUM] OpenTelemetry 1.12.0 advisories suppressed
- **Category:** Known-vulnerable dependency.
- **Root cause:** 1.12.0 carries `GHSA-4625-4j76-fww9` + `GHSA-g94r-2vxg-569j` (both moderate); the build whitelists
  `NU1902` and a code comment falsely claims 1.12.0 patched them.
- **Fix:** Bump the OTel packages to a patched line and remove the `NU1902` whitelist so the audit gate re-arms.

### 14. [MEDIUM] `SessionHistoryLookup` scans full lifetime working sets
- **Category:** Over-fetch on a hot, polled path.
- **Root cause:** The active-session "last time" reducer loads every working set across all history for the session's
  lifts, then takes the most-recent-per-lift in memory â€” unbounded in row count for high-volume lifts.
- **Fix:** Bound the scan to a recency window (with a fallback when empty) so a long-tenured user doesn't pull hundreds
  of rows on every active-session refresh.

  unreachable (all callers pass a tenant) but fragile; make it deny on null so the guard fails closed.

- **17 Resolver projections:** `ResolveExerciseNames` / `ResolveExerciseTrackingTypes` should `.Select` + `AsNoTracking`

## Documented (larger / cross-repo â€” recommended next, not done in this pass)

These are real and valuable but either break the client wire contract or are large enough to warrant their own
focused change with the clients in the loop:

- **Unify the HTTP error body (RFC-7807 `ProblemDetails` + `extensions.code`).** Today there are five shapes
  (bare string from `ToFailureResult`, `{code,message}` from auth, `ProblemDetails` from the exception handler,
  hand-rolled from the version middleware, bare string from the rate limiter). Unifying is the right end-state, **but
  the Angular + Flutter clients currently branch on bare-string-vs-`{code,message}`** (see the API wire-format memo),
  so this must land as a coordinated 3-repo change. *Approach:* make `ToFailureResult` emit `ProblemDetails` with
  `extensions["code"]`, migrate the two manual auth `BadRequest(error)` paths, update both clients' error parsing,
  ship together.
- **OpenAPI hardening.** `AddOpenApi()` is bare and dev-only: no bearer scheme, no `X-Api-Version` parameter, zero
  `ProducesResponseType`. *Approach:* add a document transformer (bearer + version header), annotate actions, serve a
  read-only spec in production.
- **HTTP-pipeline tests.** There are zero `WebApplicationFactory` tests, so status codes, `Location` headers, the
  error-body shape, and `X-Api-Version` are unverified end-to-end. *Approach:* add a thin `WebApplicationFactory`
  suite over the existing Postgres fixture for a representative endpoint slice.
- **`TimeProvider` abstraction.** 14 handlers read `DateTimeOffset.UtcNow` directly; tests capture `UtcNow` twice,
  creating a once-a-week boundary flake. *Approach:* inject .NET 10 `TimeProvider`, feed a fixed instant in tests.
- **Lower-priority polish:** correct the three broken `201` `Location` headers (or downgrade to `200`); return 404
  (not 403) for same-tenant cross-trainee access to stop the existence leak; paginate `GetTenantMembers`; tighten the
  convention tests' substring heuristics; colocate the three pipeline behaviors; `GetTenantMembers`/`Nutrition`
  mapping symmetry.
